# PRD: Algorithm Evaluation Pipeline

## Problem Statement

The current horse racing prediction system uses a single Ridge Regression model whose performance is only tracked retrospectively via monthly `PredictionScores_YYYYMM.csv` files. There is no way to compare alternative prediction algorithms against each other in a rigorous, forward-looking way before committing one to production. Adding a new algorithm today requires manually duplicating the predictor notebook, changing data loading code, and running ad-hoc experiments with no consistent methodology. The result is that algorithm improvements are slow to evaluate and hard to trust.

## Solution

Build a walk-forward evaluation pipeline that allows multiple prediction algorithms to be compared head-to-head using historical race data. The pipeline simulates prospective prediction: for each of the last 14 race days, it trains each registered algorithm on the 7 months of results prior to that day, predicts the winners for that day, and scores the predictions against known outcomes. Results are printed to the console showing per-day accuracy and a consolidated summary across all algorithms. A shared algorithm registry makes it trivial to register new algorithms and swap the active algorithm used by the daily production predictor.

## User Stories

1. As a developer, I want to run a single command to compare all registered algorithms over the last 14 race days, so that I can quickly identify which algorithm performs best before changing production.
2. As a developer, I want each algorithm to be trained on the 7 months of race results immediately prior to the day being predicted, so that the evaluation reflects realistic production conditions without data leakage.
3. As a developer, I want the evaluation to step through 14 individual race days, retraining each algorithm per day, so that I can see how performance varies day-to-day rather than just in aggregate.
4. As a developer, I want per-day accuracy and ROI printed to the console as each fold completes, so that I can monitor progress and spot variance across days while the script is running.
5. As a developer, I want a consolidated summary table printed at the end showing each algorithm's overall accuracy and ROI across all 14 days, so that I can compare algorithms at a glance.
6. As a developer, I want winner prediction accuracy to be the primary metric, so that algorithm comparisons are based on the most meaningful measure of predictive quality.
7. As a developer, I want ROI (£1 stake per predicted race) to be shown alongside accuracy as an informational metric, so that I can understand the betting value implications without it influencing algorithm ranking.
8. As a developer, I want the evaluation pipeline to write no files to disk, so that repeated runs do not pollute the data directory or interfere with the production pipeline.
9. As a developer, I want all feature engineering (categorical encoding, speed calculation, horse stats, jockey stats) to be computed in-memory from raw results files for each evaluation fold, so that there are no stale intermediate files affecting results.
10. As a developer, I want horse and jockey statistics to be computed from the same 7-month window used for training, so that the prediction features are consistent with what the model was trained on.
11. As a developer, I want all algorithms to share the same feature engineering pipeline, so that differences in scores are attributable to the algorithm and not to feature engineering choices.
12. As a developer, I want races with unknown horses or jockeys (no prior race history in the training window) to be excluded from evaluation for all algorithms, so that predictions are only made where sufficient context exists.
13. As a developer, I want each algorithm to declare its own maximum field size (maximum number of runners per race), so that algorithms can be tuned to the race sizes they handle best without affecting other algorithms.
14. As a developer, I want to register a new algorithm by adding a single entry to a central registry, so that I do not need to modify the evaluation or prediction orchestration code.
15. As a developer, I want to mark one algorithm as "active" in the registry, so that the daily production predictor automatically uses the currently preferred algorithm.
16. As a developer, I want the daily production predictor to use the same algorithm interface as the evaluation pipeline, so that an algorithm validated in evaluation behaves identically in production.
17. As a developer, I want the daily production predictor to continue reading pre-computed feature files (race features, horse stats, jockey stats, today's race cards) that are already generated by earlier pipeline steps, so that the production run time is not increased.
18. As a developer, I want the daily production predictor to continue writing `TodaysPredictions.csv`, so that the existing `validate` CLI step in `run.ps1` keeps working without changes.
19. As a developer, I want the Ridge Regression algorithm to be refactored into the new algorithm interface, so that it can be evaluated alongside new algorithms using the same pipeline.
20. As a developer, I want an XGBoost algorithm implemented under the new interface, so that there is an initial alternative to compare against Ridge Regression.
21. As a developer, I want the XGBoost algorithm to have a configurable maximum field size, so that I can tune it independently of the Ridge Regression limit.
22. As a developer, I want the algorithm interface to define a `fit` method accepting a training DataFrame, so that each algorithm can train on the same historical feature set.
23. As a developer, I want the algorithm interface to define a `predict` method accepting race card data plus horse and jockey stats, so that each algorithm can enrich and score the races it is asked to predict.
24. As a developer, I want the `predict` method to return exactly one predicted winner per race (as a RaceId/HorseId pair), so that scoring is unambiguous.
25. As a developer, I want the evaluation pipeline to clearly separate the data preparation logic from the algorithm logic, so that shared utilities can be reused by both the evaluation and production scripts without duplication.
26. As a developer, I want all algorithm implementations to live in a dedicated algorithms directory, so that they are easy to find and do not clutter the notebooks and data files.

## Implementation Decisions

### Modules to build

- **Algorithm base class / protocol** — defines the shared interface: a constructor accepting `max_horses`, a `fit(train_df)` method, and a `predict(races, horse_stats, jockey_stats)` method returning one predicted winner per race.
- **Algorithm registry** — a central list of instantiated algorithm objects, with one entry marked as the active algorithm. Lives in the algorithms package initialiser.
- **Ridge Regression algorithm** — the existing Ridge + polynomial features logic extracted from the current predictor notebook and wrapped in the new interface. Default `max_horses=10`.
- **XGBoost algorithm** — a new implementation using XGBoost with configurable `max_horses`.
- **Evaluation script (`evaluate.py`)** — orchestrates the 14-fold walk-forward loop. For each fold: loads 7 months of raw results CSVs, runs in-memory feature engineering, splits into train/eval sets, calls each algorithm's `fit` then `predict`, scores predictions, prints per-day results and a final summary.
- **Production prediction script (`predict.py`)** — replaces the converted `LinearRegressionPredictor` notebook in `run.ps1`. Reads pre-computed feature files already produced by earlier pipeline steps, calls the active algorithm's `fit` and `predict`, writes `TodaysPredictions.csv`.
- **Shared pipeline utilities** — scoring logic (accuracy, ROI calculation) and any data-preparation helpers reused by both scripts.

### Algorithm interface

- `fit(train_df)` receives a fully-featured training DataFrame (same schema as the current `Race_Features.csv`) and returns nothing.
- `predict(races, horse_stats, jockey_stats)` receives: raw race card rows for the target day, a per-horse stats DataFrame, and a per-jockey stats DataFrame. Returns a DataFrame with one row per race containing `RaceId` and `HorseId` of the predicted winner.
- Each algorithm is instantiated with a `max_horses` integer parameter. The algorithm applies this filter internally.
- The pipeline applies a `KnownHorseAndJockey == True` filter before passing data to any algorithm.

### Walk-forward fold structure (evaluate.py)

- For each of the 14 days ending yesterday (inclusive):
  - Load the 7 months of raw results files ending the day before the target date.
  - Run the full feature engineering chain in-memory: encode categoricals, calculate speed, calculate per-horse stats, calculate per-jockey stats.
  - Extract the training set (races before the target date) and evaluation races (target date only).
  - Derive current horse stats and jockey stats from the training window for use in `predict`.
  - For each registered algorithm: call `fit`, call `predict`, score against known finishing positions.
- Print per-day detail as each fold completes.
- Print consolidated summary table after all folds.

### Production prediction script (predict.py)

- Reads race features, horse stats, jockey stats, and today's race card files that are already produced by earlier `run.ps1` steps (no in-memory feature recomputation).
- Retrieves the active algorithm from the registry.
- Calls `fit` with the training features and `predict` with today's race card data and stats.
- Writes `TodaysPredictions.csv` in the existing format so the downstream `validate` CLI step continues to work.

### run.ps1 changes

- The `LinearRegressionPredictor` notebook conversion and execution steps are replaced by a call to `predict.py`.
- The `HorseStatsBuilder`, `JockeyStatsBuilder`, `FeatureAnalysis`, `validate`, and `todaysracecards` steps remain unchanged.

### Scoring

- Accuracy: percentage of predicted winners where `FinishingPosition == 1`.
- ROI: £1 stake per predicted race; winnings = decimal odds × stake for correct predictions; reported as percentage gain/loss.
- Accuracy is the ranking metric. ROI is displayed for information only.

## Testing Decisions

### What makes a good test

Tests should verify observable behaviour — what the algorithm or pipeline produces given specific inputs — not internal implementation details like which sklearn classes are used or how many intermediate variables exist. Tests should be deterministic (fixed random seeds where applicable) and should not depend on external files or network access.

### Modules to test

- **Algorithm base contract** — given a known training DataFrame and a known set of races to predict, each algorithm implementation must return exactly one `RaceId`/`HorseId` row per race, with no races duplicated or missing.
- **Ridge Regression algorithm** — verify that `fit` followed by `predict` returns plausible winners on a small synthetic dataset; verify that the `max_horses` filter excludes races above the configured field size.
- **XGBoost algorithm** — same contract tests as Ridge Regression.
- **Scoring logic** — given a fixed set of predictions and known results, verify that accuracy and ROI are calculated correctly, including edge cases (all correct, all wrong, void races).
- **Pipeline filter** — verify that the `KnownHorseAndJockey == True` filter removes the correct rows before data reaches any algorithm.

### Prior art

- Existing Python utility tests `test_data_analysis.py` and `test_data_transforms.py` demonstrate the pattern: construct a small in-memory DataFrame, call the function under test, assert on the output. New tests should follow the same pattern.
- Existing C# tests (`RaceCardParserShould`, `RacingResultParserShould`) show behaviour-focused testing without over-mocking — prefer real data fixtures over complex mocks.

## Out of Scope

- **Feature engineering improvements** — the feature set is fixed at the existing 70-column schema produced by the current `FeatureAnalysis` pipeline. Analysing or extending features is a separate activity.
- **Additional evaluation algorithms** — only Ridge Regression and XGBoost are in scope for this iteration. The registry makes it easy to add more later.
- **Live / truly forward-looking predictions** — all evaluation uses completed historical races. Predicting races before they run (using a scraped race card) is not in scope.
- **Walk-forward with more than 14 folds** — the evaluation window is fixed at 14 days. Extending this or making it configurable is not in scope.
- **Dashboard or file-based reporting** — output is console only. CSV exports, HTML reports, or notebook visualisations of evaluation results are not in scope.
- **Hyperparameter tuning** — algorithms use sensible defaults. Automated tuning (grid search, Bayesian optimisation) is not in scope.
- **Trainer statistics** — trainer stats exist in the codebase but are not currently used by the predictor. Including them is not in scope.
- **Per-race-type breakdown** — aggregate accuracy across all race types. Separate Flat/Hurdle/SteepleChase reporting is not in scope.

## Further Notes

- The `KnownHorseAndJockey == True` filter significantly reduces the evaluable race pool (approximately 6.96% of races pass this filter in the current dataset). Accuracy figures should be interpreted in this context.
- The existing Ridge Regression model filters to races with 10 or fewer runners. XGBoost may use a different limit. Both limits are configurable as constructor parameters and are applied by the algorithm internally, not by the pipeline.
- The evaluation window of 14 days was chosen to be computationally feasible while providing enough data points to distinguish algorithm performance. With retraining on each fold, the total compute is 14 × (feature engineering + model training) per run of `evaluate.py`.
- The `DecimalOdds` feature is present in `Race_Features.csv` (for completed historical races) and is used by the current Ridge Regression model. This column may not be available on truly future race cards. This is not a concern for the evaluation pipeline (which uses historical data only) but may become relevant if the system is ever extended to predict pre-race.
- No changes are made to the C# data extraction pipeline (`RaceDataDownloader`). All changes are confined to the Python `Data/` directory and `run.ps1`.
